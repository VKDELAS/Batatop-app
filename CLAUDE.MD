## 10. PadrГЈo: BotГЈo Fixo Acima do Teclado (react-native-keyboard-controller)

Usado em `app/auth/login.js` no botГЈo "Continuar" (footer fixo dos steps `email` e `name`). Documentado aqui porque o bug que resolve Г© sutil вҖ” reaplique esse padrГЈo em qualquer outra tela que precise de um botГЈo/footer que acompanha o teclado.

### HistГіrico do bug (por que nГЈo dГЎ pra resolver sГі com JS)

Duas abordagens anteriores falharam, ambas pelo mesmo motivo raiz:

1. **`KeyboardAvoidingView`**: esquecia o estado do teclado depois de um ciclo de `AppState` (app vai pra background e volta) вҖ” botГЈo ficava grudado ou nГЈo subia.

**ConclusГЈo:** qualquer abordagem baseada em resize de janela ou eventos JS do teclado Г© fundamentalmente frГЎgil no Android depois de ciclos de background/foreground. A soluГ§ГЈo definitiva Г© ler o teclado direto do nГӯvel nativo.

### A soluГ§ГЈo: `react-native-keyboard-controller`

Nada disso Г© mais necessГЎrio вҖ” o hook nativo jГЎ Г© imune a ciclos de background/foreground.

## 11. PadrГЈo: "Continuar como [nome]" / Soft-Logout

Usado no fluxo "Sim, lembrar minha conta" (`app/profile.js`) + "Continuar como {nome}" (`components/AuthBottomSheet.tsx`, Estado B). Documentado aqui porque a causa raiz Г© contra-intuitiva e fГЎcil de reintroduzir.

### O bug original

Sintoma: clicar em "Continuar como [nome]" sempre caГӯa no fallback de login (`Invalid Refresh Token: Refresh Token Not Found`), mesmo testando poucos segundos depois do logout вҖ” nГЈo era expiraГ§ГЈo de token (refresh token do Supabase **nГЈo expira por tempo**, sГі Г© invalidado por uso ou revogaГ§ГЈo).

Causa raiz real: **nГЈo existe scope de `supabase.auth.signOut()` que deixe a sessГЈo atual viva no servidor.** `scope: 'global'` (padrГЈo) revoga a sessГЈo em *todos* os dispositivos; `scope: 'local'` revoga sГі a sessГЈo atual вҖ” mas revoga. NГЈo tem meio-termo. Um comentГЎrio antigo no cГіdigo dizia que `'local'` "nГЈo revoga o token no servidor" вҖ” isso estГЎ **errado** (checar doc oficial antes de confiar nesse tipo de suposiГ§ГЈo de novo).

Ou seja: qualquer `signOut()` chamado no momento do "Sim, lembrar" matava exatamente o refresh token que o app tentaria reusar depois. Reescrever a lГіgica de refresh/setSession nГЈo resolve вҖ” o problema Г© a premissa (guardar um refresh token de uma sessГЈo que vocГӘ mesmo acabou de revogar).

### A soluГ§ГЈo: soft-logout (nГЈo existe logout de verdade aqui)

`utils/authSession.js` centraliza:
- `SOFT_LOGOUT_KEY` вҖ” flag local (`AsyncStorage`) que faz a UI **fingir** deslogado.
- `getEffectiveSession()` вҖ” retorna `null` se a flag estiver ligada, senГЈo retorna a sessГЈo real do Supabase.
- `subscribeAuthUiChange` / `emitAuthUiChange` вҖ” pub/sub em memГіria. NecessГЎrio porque ligar/desligar a flag **nГЈo** passa por `supabase.auth.onAuthStateChange` (a sessГЈo real nunca muda nesse fluxo) вҖ” sem isso, telas que jГЎ estГЈo montadas hГЎ tempo (o `_layout.js` raiz, que nunca desmonta) nГЈo saberiam que precisam reconferir o login.
- `initAuthStateListener()` вҖ” chamada **uma vez** no `_layout.js` raiz. Escuta `SIGNED_IN` real (nГЈo importa por onde: e-mail, telefone, Google, etc.) e desliga `SOFT_LOGOUT_KEY` sozinha se estiver ligada. Sem isso, um login de verdade feito enquanto o soft-logout estГЎ ativo deixava a flag presa em `'true'` pra sempre, e telas que checam a flag antes de tudo (`profile.js`) continuavam mostrando deslogado mesmo com sessГЈo nova e vГЎlida.

**"Sim, lembrar" (`profile.js`, `handleRememberAccount`):** NГғO chama `signOut()`. SГі salva `{name, email, phone}` (pra exibir "Continuar como [nome]") em `SAVED_USER_KEY`, liga `SOFT_LOGOUT_KEY`, emite `emitAuthUiChange()`. A sessГЈo real do Supabase continua 100% viva e se renovando sozinha (`autoRefreshToken: true` em `supabaseConfig.js`).

**"Continuar como" (`AuthBottomSheet.tsx`, `handleContinuarComo`):** sГі desliga `SOFT_LOGOUT_KEY`, emite o evento, confirma com `getEffectiveSession()`/`getSession()` que a sessГЈo real ainda existe, navega. Sem `refreshSession`, sem `setSession` вҖ” nГЈo tem token nenhum pra restaurar porque nada foi destruГӯdo.

**"Esquecer conta" (`profile.js`, `handleForgetAccount`):** esse sim Г© logout de verdade вҖ” `signOut()` padrГЈo (`scope: 'global'`), limpa `SAVED_USER_KEY` **e** `SOFT_LOGOUT_KEY`.

### Bug relacionado: `onAuthStateChange` pisando no soft-logout

Sintoma: `profile.js` respeitava o soft-logout certinho, mas `pedidos.js`, `addresses.js` e `_layout.js` voltavam a mostrar "logado" sozinhos depois de um tempo, mesmo com `SOFT_LOGOUT_KEY` ligada вҖ” sem nenhuma aГ§ГЈo do usuГЎrio.

Causa raiz: essas telas tinham um segundo listener que setava o usuГЎrio **direto do `session` recebido no callback** do `onAuthStateChange`, ignorando a flag:

```js
// ERRADO вҖ” ignora o soft-logout
supabase.auth.onAuthStateChange((_e, session) => setUser(session?.user ?? null));
```

A sessГЈo real do Supabase se renova sozinha em background (`autoRefreshToken: true`), o que dispara `onAuthStateChange` (ex: evento `TOKEN_REFRESHED`) mesmo com o soft-logout ligado вҖ” e esse handler reaplicava a sessГЈo real por cima. `profile.js` nunca teve esse listener direto (sГі reconfere a flag no `useFocusEffect`), por isso foi o Гәnico que ficou certo.

**CorreГ§ГЈo:** o callback do `onAuthStateChange` nunca deve usar o `session` recebido pra decidir a UI вҖ” ele serve sГі de **gatilho** pra rechamar a funГ§ГЈo que passa por `getEffectiveSession()`:

```js

  refreshEffectiveUser(); // funГ§ГЈo que chama getEffectiveSession() internamente
});
```

### Bug relacionado: `SIGNED_IN` falso na reidrataГ§ГЈo de sessГЈo (cold start) вҖ” `utils/authSession.js`

Sintoma: usuГЎrio clica em "Sim, lembrar", fecha o app de vez (mata o processo) e abre de novo вҖ” esperava ver a tela deslogada e precisar clicar em "Continuar como [nome]", mas o app jГЎ abre logado sozinho, sem nenhuma aГ§ГЈo.

Causa raiz: `initAuthStateListener()` (chamado uma vez no `_layout.js` raiz) escuta `SIGNED_IN` pra desligar `SOFT_LOGOUT_KEY` quando um login de verdade acontece (ver acima). O problema Г© que o evento `SIGNED_IN` **nГЈo dispara sГі em login interativo real** вҖ” o Supabase tambГ©m reporta `SIGNED_IN` quando reidrata a sessГЈo persistida do `AsyncStorage` ao iniciar o app, mesmo que o usuГЎrio nГЈo tenha feito nada. Esse Г© o **primeiro** evento que o listener recebe depois de assinar `onAuthStateChange`.

Sem tratar isso, todo cold-start com o soft-logout ligado disparava esse `SIGNED_IN` de reidrataГ§ГЈo, o listener achava que era um login de verdade, desligava a flag sozinho, e a pessoa aparecia logada de novo sem nunca ter tocado em "Continuar como".

**CorreГ§ГЈo:** ignorar explicitamente o primeiro evento recebido depois de assinar o listener вҖ” ele Г© sempre a reidrataГ§ГЈo/estado atual, nunca uma transiГ§ГЈo real:

```js
export function initAuthStateListener() {
  if (listenerInitialized) return;
  listenerInitialized = true;

  // O primeiro evento apГіs assinar Г© sempre a reidrataГ§ГЈo da sessГЈo do
  // AsyncStorage no cold start, nГЈo um login real вҖ” precisa ser ignorado.
  let isInitialEvent = true;

  supabase.auth.onAuthStateChange(async (event, session) => {
    if (isInitialEvent) {
      isInitialEvent = false;
      return;
    }

    if (event === 'SIGNED_IN' && session) {
      const soft = await AsyncStorage.getItem(SOFT_LOGOUT_KEY);
      if (soft === 'true') {
        await AsyncStorage.setItem(SOFT_LOGOUT_KEY, 'false');
        emitAuthUiChange();
      }
    }
  });
}
```

# CLAUDE.md

Behavior guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" вҶ’ "Write tests for invalid inputs, then make them pass"
- "Fix the bug" вҶ’ "Write a test that reproduces it, then make it pass"
- "Refactor X" вҶ’ "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] вҶ’ verify: [check]
2. [Step] вҶ’ verify: [check]
3. [Step] вҶ’ verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

---

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.
